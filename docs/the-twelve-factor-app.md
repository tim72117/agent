## 背景

在現代，軟體通常被當作一種服務來交付，這種軟體稱為網路應用程式（web apps），或稱軟體即服務（software-as-a-service, SaaS）。十二要素應用程式（twelve-factor app）是一套方法論，用來打造具備以下特性的 SaaS 應用程式：

- **使用宣告式（declarative）格式**來設定自動化流程，讓新加入專案的開發者能以最少的時間與成本上手；
- **與底層作業系統之間有乾淨的契約（clean contract）**，在各種環境之間提供最大的可攜性（portability）；
- **適合部署在現代雲端平台**上，讓伺服器與系統管理相關的需求降到最低；
- **將開發環境與正式環境之間的落差降到最小**，支援持續部署（continuous deployment）以獲得最大的敏捷性；
- 能夠**在不需要大幅改變工具、架構、或開發實務的前提下擴展規模（scale up）**。

十二要素方法論適用於任何使用任何程式語言撰寫、搭配任何組合的後端服務（資料庫、佇列、記憶體快取等）的應用程式。

這套方法論的貢獻者，都曾親身參與過數百個應用程式的開發與部署，並透過 [Heroku](https://www.heroku.com/) 平台，間接見證、參與過數千個應用程式的開發、運作、與擴展過程。這份文件整理了我們對於各式各樣 SaaS 應用程式在實務上所觀察到的廣泛經驗與心得。它試圖整合出**打造 SaaS 應用程式的一套最佳實務**，並特別著重於以下幾個面向背後的動能（dynamics）：

- 應用程式**有機成長**的過程——也就是程式碼庫（codebase）隨著時間演進的過程；
- **開發者之間的協作**——當多位開發者同時在一份程式碼庫上工作時；
- 避免**軟體侵蝕（software erosion）的成本**——也就是避免專案在生命週期中逐漸腐化、難以維護的代價。

我們撰寫這份文件的動機，是希望能提高大家對這些議題的意識，**為討論這些問題提供一套共同的詞彙**，並提供一套解決這些問題的概念性框架，同時搭配特定的術語命名。這樣的呈現形式，某種程度上是受到 Martin Fowler 所著、關於企業架構模式（enterprise application architecture patterns）等書籍的啟發。

## I. 程式碼庫（Codebase）

### 一份程式碼庫，透過版本控制追蹤，可以有多個部署

十二要素應用程式**必定**使用某種版本控制系統來追蹤，例如 Git、Mercurial、或 Subversion。追蹤程式碼庫的一份拷貝，稱為一份 code repo（程式碼庫），或簡稱 repo。

**程式碼庫（codebase）**指的是：

- 在使用集中式版本控制系統的情況下，就是「唯一那一個 repo」；
- 在使用分散式版本控制系統（例如 Git）的情況下，指的是「共享同一個根提交（root commit）的一組 repo」。

**應用程式與程式碼庫之間，永遠是一對一的對應關係：**

- 如果存在多個程式碼庫，那就不是一個應用程式——而是一個**分散式系統（distributed system）**。分散式系統裡的每一個元件，都是一個各自獨立的應用程式，而分散式系統裡的每一個元件，都能個別遵循十二要素方法論。
- 多個應用程式共用同一份程式碼，是違反十二要素原則的行為。解決辦法是把共用的程式碼拆分成獨立的函式庫（library），再透過該語言的套件管理機制（dependency manager）將其納入。

**一個應用程式雖然只有一份程式碼庫，但可以同時存在多個部署（deploy）。** 一個「部署」指的是某一個正在運作中的應用程式實例。這通常是一個正式環境（production）站台；此外，每一個 staging（預備／測試）環境，也算是一個部署。此外，每一位開發者在本機的開發環境，也算是一個部署（雖然性質有所不同）。

程式碼庫在所有部署之間都是相同的，儘管每個部署可能運行著不同版本的程式碼。舉例來說，開發者的本機環境裡，可能有一些尚未部署到 staging 環境的提交（commit）；staging 環境裡，也可能有一些尚未部署到正式環境的提交。但這些部署所使用的，都是同一份程式碼庫——這也是為什麼它們可以被視為同一個應用程式的多個不同部署。

## II. 依賴（Dependencies）

### 明確地宣告並隔離依賴套件

大多數的程式語言，都提供了一套用來散布支援函式庫（support library）的套件系統，例如 Perl 的 CPAN、Ruby 的 RubyGems。透過套件管理工具安裝的函式庫，可以是**系統層級的安裝**（有時稱為「site packages」），也可以是**限定在應用程式目錄範圍內、與特定應用程式綁定**的安裝（有時稱為「vendoring」或「bundling」）。

**十二要素應用程式，絕不會依賴「系統層級套件必然存在」這個假設。** 它會透過一份依賴宣告清單（dependency declaration manifest），完整且精確地宣告所有的依賴套件，且不留任何隱含的模糊空間。此外，在執行的當下，它會使用一套**依賴隔離工具（dependency isolation tool）**，確保不會有任何隱含的依賴，從周遭的系統環境中悄悄滲透進來。這一套「宣告 + 隔離」的完整規範，必須同時套用在正式環境與開發環境上。

舉例來說：Ruby 的 [Bundler](http://bundler.io/) 透過 `Gemfile` 這份清單檔案來做依賴宣告，並透過 `bundle install` 隔離安裝所有的 gem；執行時再搭配 `bundle exec` 來確保程式只會存取到 `Gemfile` 裡明確列出的那些 gem，隔絕任何系統既有的 gem。Python 有兩種對應的工具分別做這兩件事：用 [Pip](http://www.pip-installer.org/en/latest/) 做依賴宣告（搭配 `requirements.txt` 檔案），再搭配 [Virtualenv](http://www.virtualenv.org/en/latest/) 做隔離。就算是 C 語言，也有對應的做法：用 [Autoconf](http://www.gnu.org/s/hello/manual/autoconf/) 做依賴宣告，再透過靜態連結（static linking）搭配類似 [Bundler](http://bundler.io/) 的工具鏈來做隔離。無論使用哪一種工具鏈，「宣告」與「隔離」這兩件事都必須**同時**做到——只做其中一件是不夠的。

明確宣告依賴的一項好處在於：這讓新加入專案的開發者，能夠更輕鬆地上手。一位想要開始參與這個應用程式開發的新手開發者，只需要在自己的開發機器上安裝該應用程式所使用的程式語言執行環境（language runtime），以及對應的依賴管理工具（dependency manager）即可。他只要執行該語言依賴管理工具的建置指令（build command），就能把所有需要的東西一次裝齊——例如 Ruby 的 `bundle install`、Clojure 的 `lein deps`。整個過程不需要額外安裝任何系統套件，也不需要任何額外的系統設定步驟，就能讓應用程式在本機端跑起來，而且這個過程是**確定性（deterministic）**的：只要輸入相同的依賴宣告清單，任何時候、任何機器上跑出來的依賴組合都會一致。

這項特性同時對正式環境的部署也很有價值：一個乾淨的系統，透過語言的套件管理器，就能組出應用程式運作所需要的完整依賴——不需要任何額外的系統套件安裝步驟。

十二要素應用程式，也不會假設任何系統工具是必然存在的。舉例來說，就算應用程式在執行 shell 指令時剛好會用到 [ImageMagick](http://www.imagemagick.org/) 或 `curl`，也不能想當然爾地認為這些系統工具，在應用程式將來所部署的每一個環境裡都一定存在，或都具有應用程式所預期的相容版本。如果應用程式真的需要透過 shell 呼叫某個系統工具，這個工具就應該被 vendor（附帶）進應用程式本身裡。

## III. 設定（Config）

### 把設定值存放在環境變數裡

一個應用程式的**設定（config）**，指的是所有「在不同部署之間，很可能會有所不同」的東西——這包括：

- 資料庫、Memcached、以及其他後端服務（backing service）的連線資訊；
- 像是 Amazon S3、Twitter 這類第三方服務的憑證；
- 每一個部署專屬的一些設定值，例如該部署所屬的 hostname（主機名稱）。

我們有時會看到一些應用程式，把設定值寫死成程式碼裡的常數（constant）——這正是十二要素方法論所要求嚴格避免的做法。**設定值，會在不同的部署之間有所不同，但程式碼不會。**

一個簡單的判斷準則是：檢查應用程式的設定值是否嚴格地與程式碼區隔開來。有一個很實用的驗收測試：**這份程式碼庫，是否可以在「不會洩漏任何機密憑證」的前提下，隨時被開源（open source）出去？** —— 值得注意的是，這個「開源與否」的判準所涵蓋的範圍，比十二要素應用程式所定義的「設定（config）」還要更廣一些：舉例來說，一個應用程式內部使用的路由規則檔案（例如 Rails 的 `config/routes.rb`），並不算是十二要素方法論所定義的「設定」，因為即使把這份檔案公開，也不會有任何機密洩漏的風險——而且無論是哪一個部署，這份檔案的內容都應該是完全相同的。

十二要素方法論建議把設定值存放在**環境變數（environment variables，簡稱 env vars 或 env）**裡。環境變數很容易在不同的部署之間進行變更，卻不需要更動任何一行程式碼；而且相較於設定檔（config file），它幾乎不可能被不小心提交（commit）進版本控制系統裡。與 Java 那種傳統的設定檔做法（例如 `config/database.yml`，這類設定檔通常不會被統一放在同一個目錄下）相比，環境變數是一種與程式語言、與作業系統都無關（language- and OS-agnostic）的標準做法。此外，環境變數也讓每一項設定值，都能被獨立地進行細粒度（granular）的控制與調整，不需要把它們統一歸類進某個籠統的「環境（environment）」分組裡。

十二要素方法論裡的「環境」，指的其實不是傳統意義上開發者所熟悉的那種、以名稱區分的環境群組（例如 Rails 裡常見的 `development`、`test`、`production` 這幾種）。取而代之的做法是：把每一個部署所需要調整的設定值，各自獨立地去做管理。這種模式能讓應用程式在規模持續成長的過程中，更加平順地擴展——因為隨著應用程式規模擴大、部署數量增加，開發團隊自然會需要建立起越來越多的部署環境，例如專門用來因應某個大客戶而建立的 staging 環境，或是專門用來準備上線的 pre-production 環境。如果設定值是用「具名環境」這種形式來分組的話，這些新增的部署環境很快就會迫使開發團隊，去新增一堆特殊的、一次性用途的具名環境（例如 `joes-staging`），這種做法會讓整個設定管理迅速變得混亂、難以維護。

## IV. 後端服務（Backing services）

### 把後端服務都當作是「附加資源（attached resources）」

一個**後端服務（backing service）**，指的是應用程式在正常運作過程中，會透過網路來消費（consume）的任何服務。常見的例子包括：

- 像 [MySQL](https://www.mysql.com/) 或 [CouchDB](http://couchdb.apache.org/) 這類資料庫服務；
- 像 [RabbitMQ](http://www.rabbitmq.com/) 或 [Beanstalkd](https://beanstalkd.github.io/) 這類訊息佇列（message queuing）系統；
- 像 [Postfix](http://www.postfix.org/) 這類透過 SMTP 收發電子郵件的服務；
- 像 [Memcached](http://memcached.org/) 這類快取（caching）系統。

後端服務，跟應用程式所執行的一般程式碼一樣，都是透過網路來存取的資源；它們消費的是資料，或提供其他功能。舉例來說：像 MySQL 這類資料庫服務，通常是由該應用程式部署團隊自行管理（在同一台伺服器上、或另外一台獨立的伺服器上）；也可能是由第三方——例如透過 Amazon RDS 或 Heroku Postgres 這類「資料庫即服務（database-as-a-service）」供應商——來代為管理。像 SMTP 這類寄信服務，也一樣：可能是由部署團隊自行架設的 Postfix 伺服器，也可能是委外交給第三方（例如 Postmark）來代管。第三方後端服務的另外一些例子，還包括像是 Amazon S3 這類的資產儲存（asset storage）服務、像是 New Relic 或 Loggly 這類的分析服務、像是 Twitter、Google Maps、Last.fm 這類透過 API 存取的消費者服務。

**十二要素應用程式的程式碼，完全不會去區分「本地端服務」跟「第三方服務」這兩者。** 對應用程式來說，這兩種服務都只是「附加資源（attached resource）」——都是透過設定裡的一個 URL 或其他形式的憑證，來進行存取。一個十二要素應用程式的任何一個部署，都應該能夠在**不需要更動任何一行程式碼**的前提下，把一份本地端的 MySQL 資料庫，替換成另一份由第三方所管理的資料庫服務（例如 Amazon RDS），而完全不需要更動應用程式的程式碼——唯一需要更動的，只有存放在設定裡的那個資源連線資訊（resource handle）。

每一個不同的後端服務，都算是一項**資源（resource）**。舉例來說，一份 MySQL 資料庫是一項資源；如果為了做資料庫分片（sharding）而使用了兩份彼此獨立的 MySQL 資料庫，那麼從十二要素應用程式的角度來看，這就算是**兩項**彼此獨立的資源。十二要素應用程式會把這些資料庫，都當作是**附加資源（attached resources）**來處理——這樣的描述方式，也暗示了這些資源與它所服務的部署之間，是一種鬆散耦合（loosely coupled）的關係。

一項資源，可以在應用程式的任何一個部署裡，被隨時「附加（attach）」上去、或「卸除（detach）」下來。舉例來說，假設應用程式的其中一份資料庫，因為硬體問題而發生異常，這時資料庫管理員可能需要啟用一份新的資料庫伺服器來取代它——並從最近一次的備份還原資料。在這個過程中，原本那份出問題的資料庫資源，可以先被卸除，接著再把那份新建的、還原完成的資料庫，重新附加上去——而整個過程，完全不需要更動應用程式的任何一行程式碼。

## V. 建置、發布、運行（Build, release, run）

### 嚴格區分「建置（build）」與「運行（run）」這兩個階段

一份程式碼庫，會經過以下的轉換過程，變成一個（非開發環境下的）部署：

1. **建置階段（build stage）**：這是一個把程式碼庫轉換成一個「可執行套件包（build）」的過程。這個建置階段，會取用某一個特定版本（也就是某個特定的 commit）的程式碼，抓取並打包（vendor）好所有它所依賴的套件，並把程式碼編譯（compile）成執行檔與各類資產檔案（asset）。
2. **發布階段（release stage）**：這個階段，會把上一步驟所產生的建置成果（build），再結合上這個部署當下所對應的設定值（config），組合出一份「發布版本（release）」。這份發布版本，包含了建置成果與設定這兩者，已經完全準備就緒、隨時可以在執行環境（execution environment）裡被啟動、運行。
3. **運行階段（run stage）**（有時也稱為「runtime」）：這個階段，會針對某一份特定的發布版本，在應用程式當下所屬的執行環境裡，啟動起應用程式的一組（或多組）程式（process）。

**十二要素應用程式，會嚴格地把「建置」、「發布」、「運行」這三個階段區分開來。** 舉例來說，若在應用程式已經進入運行階段之後（也就是程式已經在運作中），才試著去更動它的程式碼，這樣的做法是完全不可行的——因為這樣一來，執行中的程式，就沒辦法把這個異動，同步回饋到建置階段去。

有許多部署工具，都設計成專門用來管理應用程式的程式碼，並負責在部署的當下，同時執行建置與發布這兩個階段。與此相對，也有一些專門用來執行運行階段的工具，例如 [Chef Solo](https://docs.chef.io/chef_solo.html)。

每一份發布版本，都應該永遠帶有一個獨一無二的發布識別碼（release ID），可以用時間戳記（例如 `2011-04-06-20:32:17`），也可以用一個持續遞增的編號（例如 `v100`、`v101`）來表示。發布版本，一旦被建立出來之後，就**不應該再被更動**——任何異動，都必須透過建立一份**全新**的發布版本來完成。

建置階段，可以被設計得相對複雜一些，因為它是在部署的過程中主動觸發的、有開發者在旁監督其執行結果；但相對地，運行階段，則應該被設計得盡可能地簡單——因為在實際運作的正式環境裡，運行階段的失敗，很可能發生在半夜裡、沒有任何開發者在旁邊盯著的狀況下（例如遭遇伺服器重開機、或是因程式當機而觸發自動重啟等狀況）。也因此，建置相關的工具，完全不應該在運行階段裡被牽涉進來——這代表，任何形式的程式碼編譯步驟，或是任何需要修改設定值的動作，都必須在建置階段就先完成、確定下來。

發布管理工具，通常也都提供了「回滾（rollback）」的能力，讓應用程式能回到先前的某一份發布版本。舉例來說，[Capistrano](https://github.com/capistrano/capistrano) 這套部署工具，就是把過去每一份發布版本，都存放在目標伺服器上的一個子目錄裡，讓回滾這個動作，可以透過建立一個符號連結（symlink），直接指向較早之前的那一份發布版本來完成。

## VI. 程式（Processes）

### 把應用程式，當作一個（或多個）無狀態（stateless）程式來執行

應用程式，會在執行環境裡，以一個（或多個）**程式（process）**的形式來運作。

在最簡單的情境下，程式碼本身，其實就只是一支獨立的腳本（script），而執行環境，則是開發者本機上，安裝好的某個特定版本的程式語言執行環境（language runtime）；開發者只要在本機命令列上，直接執行這支腳本，就能啟動這個程式（例如執行 `python my_script.py`）。而在光譜的另外一端，一個正式上線、已經上了規模的應用程式，則可能會由許多**程式類型（process type）**所組成，而每一種程式類型底下，又可能同時運行著多個程式實例（instance）。

**十二要素方法論裡的程式，是完全無狀態（stateless）、且彼此之間不共享任何東西（share-nothing）的。** 任何需要被長久保存下來的資料，都必須被存放在某個**具備狀態（stateful）的後端服務**裡——這通常會是一份資料庫。

程式的記憶體空間、或它的檔案系統，都可以在一次的交易（transaction）過程中，被拿來當作一次性的短暫快取空間使用——舉例來說，先把一份很大的檔案下載下來，進行處理，再把處理完的結果，寫回資料庫裡，這樣的用法是沒問題的。然而，十二要素應用程式**絕對不能**假設任何存放在程式記憶體、或檔案系統裡的東西，在下一次的請求（request）或工作（job）到來時，仍然會繼續存在——因為當有多個程式同時在運作時，接下來進來的請求，很有可能會被路由（route）到跟前一次不同的另外一個程式去處理。就算在只有單一一個程式在運作的情況下，重新啟動（無論是因為一次新的程式碼部署、或是因為設定值的變更、抑或是因為執行環境本身把該程式重新調度到另外一台實體機器上運作），也通常會把這個程式裡，原本存放在本機端的所有狀態，全部清空重置。

有一些像是 [django-assetpackager](http://code.google.com/p/django-assetpackager/) 這類的資產打包工具，會把編譯完成的資產（asset），快取存放在本機的檔案系統裡。十二要素應用程式，更偏好在建置階段（build stage），就先把這類資產都編譯、打包完成——而不是等到運行階段（run stage）才進行。像是 [Jammit](http://documentcloud.github.com/jammit/) 這類的資產打包工具、或是 Rails 的資產管線（asset pipeline），都可以被設定成，在建置階段就先完成這樣的打包過程。

有些網路系統，會依賴所謂的「黏性 session（sticky session）」機制——也就是把使用者 session 裡的一些資料，快取存放在某一個程式的記憶體裡，並期待該使用者未來的請求，都能持續被路由到同一個程式上去處理。黏性 session，是明顯違反十二要素方法論的一種做法，應該要盡量避免使用。Session 相關的狀態資料，適合被存放在具備自動過期機制（time-expiration）的資料儲存系統裡，例如 [Memcached](http://memcached.org/) 或 [Redis](http://redis.io/)。

## VII. 埠號綁定（Port binding）

### 透過埠號綁定（port binding），對外提供服務

網路上的應用程式，有時候會被建置在某個網頁伺服器容器（webserver container）裡面來運作。舉例來說，PHP 程式，有可能是被放在 [Apache HTTPD](http://httpd.apache.org/) 底下、當作一個模組（module）來運作；又或者，Java 程式，有可能是被放進 [Tomcat](http://tomcat.apache.org/) 這類的容器裡來運作。

**十二要素應用程式，則是完全自我完備（self-contained）的**，它完全不會去依賴任何執行環境中，事先架設好的網頁伺服器，就能對外提供服務。網路上的應用程式，會透過**埠號綁定（port binding）**這樣的機制，來對外公開 HTTP 這樣的服務——並在該埠號上，監聽（listen）著送進來的請求。

在本機的開發環境裡，開發者透過類似 `http://localhost:5000/` 這樣的網址，來連上自己的應用程式。而在正式環境裡的部署當中，則會有一層路由層（routing layer），負責把外部對外公開網域名稱（domain name）所送來的請求，轉發（route）到那些已經完成埠號綁定的網路程式上。

這樣的做法，通常是透過把某個網頁伺服器函式庫（webserver library），直接透過依賴宣告的方式，納入應用程式本身來實現——例如 Python 的 [Tornado](http://www.tornadoweb.org/)、Ruby 的 [Thin](http://code.macournoyer.com/thin/)、Java（以及其他基於 JVM 語言）的 [Jetty](http://www.eclipse.org/jetty/)，甚至也包括一些以其他語言撰寫、且提供 HTTP 為原生服務的執行環境，例如 [Node.js](http://nodejs.org/) 的 [Express](http://expressjs.com/) 框架。這一切都完全在應用程式自身的**使用者空間（userspace）**裡完成，也就是應用程式自己的程式碼裡——完全不需要依賴任何注入型（injected）、或外部設定好的網頁伺服器。

值得注意的是，這種模式，並不僅僅侷限於 HTTP 服務——幾乎任何一種**基於伺服器（server-like）的服務**，都可以透過在某個埠號上進行監聽，來對外提供服務，例如：[ejabberd](http://www.ejabberd.im/)（透過 [XMPP](http://xmpp.org/)），或是 [Redis](http://redis.io/)（透過自己專屬的通訊協定）。

此外，值得注意的是，這種「透過埠號綁定來對外提供服務」的做法，還代表著：**一個應用程式，可以成為另外一個應用程式的後端服務（backing service）**——只要把提供服務那一方應用程式的 URL（也就是它所綁定的埠號），透過設定值的方式，提供給接收服務的那一方應用程式即可。

## VIII. 並行（Concurrency）

### 透過程式模型（process model），做水平擴展（scale out）

任何一支電腦程式，一旦被執行起來，就會對應到一份或多份的作業系統程式（process）。網路上的應用程式，長期以來，一直都以各式各樣不同的形式，來呈現這種「程式（process）」的概念。舉例來說，PHP 程式，是以子程式（child process）的形式存在，由 Apache 隨著送進來的請求量，動態地產生、銷毀；Java 程式，則正好相反：JVM 提供了一個巨大的單一超級程式（uber-process），透過內部的執行緒（thread）機制（有時搭配類似 [Java Virtual Machine 的執行緒排程](http://java.sun.com/docs/hotspot/threads/threads.html) 這類的手段），來管理內部的並行（concurrency）。

**在十二要素應用程式裡，程式是一個「一等公民（first class citizen）」。** 十二要素應用程式裡的各個程式，深受 Unix 作業系統下、服務型常駐程式（daemon）的啟發，設計上有意識地借鏡了這樣的模型。開發者可以透過這樣的模型，來安排應用程式，該如何去因應各種不同的工作負載（workload）——舉例來說，HTTP 請求，可以交由一個網路程式（web process）來處理；而像是長時間運行的背景任務（background task），則可以交由另一個工作程式（worker process）來處理。這樣的設計，同時也不排除個別程式內部，各自去處理自己內部的多工（multiplexing）需求——也就是說，程式內部依然可以透過執行緒（thread），來處理該程式自身內部的並行需求，或是透過像是 [EventMachine](https://github.com/eventmachine/eventmachine)、[Twisted](http://twistedmatrix.com/trac/)、[Node.js](http://nodejs.org/) 這類的事件驅動（event-driven）模型，來達成類似的效果。然而，若要讓應用程式能做到**水平擴展（scale out）**——也就是把負載，分散到多台實體機器上去運作——這就必須仰賴前面所提到的那種「跨越多個程式」的擴展模型（process model）了。

十二要素應用程式的模型下，開發者能夠透過這樣的程式模型，優雅地把應用程式的工作負載擴展開來。網路程式所要處理的 HTTP 請求，可以交由一個 [Ruby unicorn 主控程式（master process）](http://unicorn.bogomips.org/) 底下的多個 unicorn worker 程式來共同分攤；而像是佇列裡的一項項工作（job），也可以交由 Python 底下的 [Celery](http://www.celeryproject.org/) 工作程式來共同分攤處理。這樣的模型並不排除個別程式，各自去處理自己內部的多工需求——也就是用執行緒、或是像 [EventMachine](https://github.com/eventmachine/eventmachine)、[Twisted](http://twistedmatrix.com/trac/)、[Node.js](http://nodejs.org/) 這類事件驅動的做法。然而，一個個別的虛擬機器，其規模終究有其極限，所以應用程式必須具備能力，跨越多台實體機器，同時運行多個程式。

這種程式模型，在需要處理程式的擴展需求時，真正能夠大放異彩。這種以程式為基礎、share-nothing（彼此不共享任何東西）的水平擴展模型，也讓十二要素應用程式，能夠**優雅地應對突然湧入的大量流量或工作負載**——只要視需求，動態新增更多的程式即可。此外，這種模型，也讓應用程式擁有極高的**韌性（robustness）**——即使某一個運作中的程式突然崩潰，也不會影響到整個應用程式的其他部分，程式管理器（process manager）能很快地重新啟動它。

**程式永遠不應該讓自己常駐化（daemonize），也不應該自行寫入 PID 檔案。** 開發者應該仰賴作業系統內建的程式管理器（例如 [systemd](https://www.freedesktop.org/wiki/Software/systemd/)、分散式的雲端平台服務程式管理器、或是像 [Foreman](http://blog.daviddollar.org/2011/05/06/introducing-foreman.html) 這類的開發工具），來負責管理程式輸出的串流（output stream）、應對程式崩潰後的重啟、以及使用者所觸發的重新啟動或關閉操作。

## IX. 可拋棄性（Disposability）

### 透過快速啟動、優雅關閉，將程式的韌性最大化

**十二要素應用程式裡的程式，是「可拋棄（disposable）」的**——也就是說，它們可以在幾乎任何一個時間點，被啟動、或被停止。這種特性，有助於快速、彈性地進行水平擴展（scale out）、快速地部署程式碼或設定值的變更，並提升正式環境部署下的韌性（robustness）。

**程式應該盡可能把啟動時間，縮到最短。** 理想的狀況下，一個程式從被觸發啟動的那一刻起，到真正進入可以開始接受請求或工作的狀態，所花費的時間，應該只需要幾秒鐘。啟動時間短，能為部署（deploy）以及擴展（scale up）這兩件事，都帶來更高的敏捷性（agility）；此外，它也能提升系統的韌性，因為程式管理器（process manager），能夠在需要的時候，更輕鬆自如地把程式搬遷（move）到不同的實體伺服器上運作。

**程式在收到來自程式管理器的 SIGTERM 訊號時，應該要能夠優雅地結束運作（graceful shutdown）。** 對於一個網路程式（web process）來說，所謂的優雅關閉，指的是：停止繼續監聽（listen）服務所在的埠號（也就是拒絕接受任何新進來的請求）、讓當下正在處理中的請求，能夠繼續完成處理，接著才真正結束程式（exit）。這個做法背後，隱含著一個前提假設：所有透過該埠號送進來的 HTTP 請求，本身的處理時間都相對短暫（最多幾秒鐘），又或者，在遇到類似長輪詢（long-polling）這類的用戶端請求、連線意外中斷的情況下，用戶端會自動嘗試重新建立連線。

至於工作程式（worker process）的優雅關閉，則可以透過把當下正在處理中的工作（job），重新交還（return）給工作佇列（work queue）來實現。舉例來說，在 [RabbitMQ](http://www.rabbitmq.com/) 裡，工作程式可以主動送出一個 `NACK` 訊號；而在 [Beanstalkd](https://beanstalkd.github.io/) 裡，只要工作程式的連線中斷，尚未完成的工作，就會被自動地、重新放回佇列裡。至於那些依賴鎖定機制（lock）的系統，則需要注意——例如 [Delayed Job](https://github.com/collectiveidea/delayed_job)——工作程式在結束運作前，也需要記得把它原本持有的那個鎖，給釋放（release）掉。無論如何，任何一項工作，都應該被設計成，具備可重入性（reentrant）——也就是說，透過交易機制（transaction）、或是透過小心的操作設計，確保這項工作，就算被重複執行多次，也不會產生任何副作用（也就是所謂的**冪等（idempotent）**特性）。

**程式，也需要具備能力，能夠承受來自硬體層級、突如其來的、非計畫性的（sudden, unexpected）終止，而不至於整個系統崩潰。** 儘管這種狀況，相較於前面所提到的、收到 SIGTERM 訊號而優雅關閉的情境，發生的頻率較低——但這樣的狀況，終究還是有可能會發生。舉個較為極端的例子：如果應用程式所在的那台底層實體硬體，本身發生故障，這時就有可能會發生這種非計畫性的終止情況。針對這樣的情境，一個推薦的做法是，使用一套具備韌性的佇列系統，例如 [Beanstalkd](https://beanstalkd.github.io/)——因為它能在用戶端連線意外中斷、或是超過設定的逾時（timeout）時間之後，自動地把原本交付出去、尚未完成的工作，重新放回佇列裡。**由於這種非計畫性、突發式的終止情況，終究還是有可能會發生，因此，十二要素應用程式的設計理念，是把「崩潰後也能安全恢復（crash-only design）」，當作是理所當然的設計前提之一。**

## X. 開發環境／正式環境的對等性（Dev/prod parity）

### 讓開發環境、staging 環境、與正式環境，都盡可能地保持相似

長久以來，在軟體開發的世界裡，開發環境（也就是一位在職開發者，在他自己的本機端進行即時開發的環境）與正式環境（也就是最終使用者，實際存取應用程式的環境）之間，向來都存在著明顯的落差，通常會呈現在以下幾個面向：

- **時間落差（time gap）：** 一位開發者所撰寫的程式碼，可能要等上好幾天、好幾週、甚至好幾個月，才會真正被部署上線。
- **人員落差（personnel gap）：** 撰寫程式碼的開發者，與負責部署程式碼的維運人員（operation engineer），往往是兩批不同的人。
- **工具落差（tools gap）：** 開發者所使用的技術堆疊（stack），有可能跟正式環境所使用的技術堆疊，存在著相當程度的落差——舉例來說，開發環境用的是 Nginx、SQLite、與 OS X，而正式環境用的，卻是 Apache、MySQL、與 Linux。

**十二要素應用程式，是設計來把上述這幾種落差，全部降到最低的：**

- **讓「時間落差」降到最低**：讓開發者所撰寫的程式碼，能在數小時、甚至只要幾分鐘之內，就完成部署。
- **讓「人員落差」降到最低**：讓撰寫程式碼的開發者，能夠緊密地參與程式碼的部署過程，並密切觀察程式碼在部署上線之後的實際運作狀況。
- **讓「工具落差」降到最低**：讓開發環境與正式環境，能夠盡可能地保持一致。

以下這張表格，總結了上述這些原則，實際套用到各種面向時，所呈現出來的具體差異：

| | 傳統做法 | 十二要素應用程式 |
| --- | --- | --- |
| 部署之間的時間間隔 | 數週 | 數小時 |
| 程式碼撰寫者 vs. 部署者 | 不同的人 | 同一群人 |
| 開發環境 vs. 正式環境 | 彼此不同 | 盡可能保持相似 |

在應用程式所使用的所有後端服務裡（例如它自己的資料庫、佇列系統、記憶體快取等），**「後端服務」這個項目**，是特別容易在開發環境與正式環境之間，出現明顯落差的一個環節。有些程式語言，會提供一些能簡化本機開發過程的函式庫，例如：讓開發者能在本機使用一份輕量型的 SQLite 資料庫，而正式環境上，則使用一份較為健全（heavyweight）的資料庫服務，例如 PostgreSQL；又或者，讓開發者在本機端使用一套記憶體內部（in-memory）的快取機制，而正式環境上，則改用 Memcached。

**十二要素方法論，強烈反對這種「開發環境與正式環境使用不同後端服務」的做法**，因為這會讓兩者之間，出現微妙的不相容之處，讓程式在開發環境裡運作正常、卻在正式環境裡發生看似莫名其妙的失敗狀況。這類不相容的問題，會大幅提升持續部署（continuous deployment）過程中的摩擦成本。從整個應用程式的生命週期來看，這種因小失大所付出的代價，遠遠超過了「在本機端額外多裝一套跟正式環境一致的後端服務」，所帶來的那麼一點點便利性。

值得慶幸的是，如今在輕量化的本機端服務裡，要維持這種一致性（parity），已經變得越來越簡單、也越來越可負擔了：

- 現代的套件管理工具，例如 Linux 底下的 [Homebrew](http://mxcl.github.com/homebrew/)、[apt-get](https://help.ubuntu.com/community/AptGet/Howto)，讓開發者能夠輕鬆地在本機端，安裝好各式各樣的後端服務。
- 像是 [Chef](http://www.opscode.com/chef/) 或 [Puppet](http://puppetlabs.com/) 這類的自動化組態配置（provisioning）工具，搭配像 [Vagrant](http://vagrantup.com/) 這類的輕量型虛擬機器管理工具，讓開發者能夠在本機端的虛擬機器裡，跑出一套跟正式環境幾乎一模一樣的環境。
- Docker、以及類似的容器化（containerization）技術，讓開發者能透過同一份容器映像檔（image）的定義，在本機端跟正式環境裡，都跑出完全相同的執行環境。

在一個應用程式的所有部署裡（無論是開發者自己本機端的環境、還是正式環境），都應該使用**同一種類型、同一個版本**的後端服務。儘管在不同的後端服務之間切換，會帶來一些不小的挑戰，但這些挑戰，都應該透過確保開發環境與正式環境之間的對等性（parity），來加以克服——而不是走回頭路，退回到使用不同類型後端服務的老路上去。至於能夠支援多種後端服務的可調適介面層（adapter），依然有它存在的價值——只不過，它的目的，應該是為了讓應用程式能夠更輕鬆地在不同的後端服務之間進行**遷移（migrate）**，而不是為了讓應用程式，能夠同時支援多種不同的後端服務。

## XI. 日誌（Logs）

### 把日誌，當作是事件串流（event stream）來處理

**日誌（log）**，讓一個正在運作中的應用程式，其行為變得可以被觀察、被理解。在伺服器端的環境裡，日誌通常會被寫入到某個檔案裡（也就是所謂的「logfile」），並隨著時間的推移，不斷地被附加寫入（append）到這份檔案的末端；但這種「寫入到某個檔案」的輸出格式，其實只不過是日誌眾多可能呈現形式裡的**其中一種**而已。

**日誌，其實是「一連串按時間排序、彙整而成的事件串流」**——這些事件，來自於這個應用程式裡，所有正在運作中的程式、以及所有它所使用的後端服務，各自輸出串流（output stream）裡的內容。日誌的原始形式，通常是每一行文字，代表一個事件（不過，也存在一些例外，例如某些程式語言在拋出例外狀況時，堆疊追蹤（stack trace）的內容，可能會橫跨好幾行）。日誌本身，並不具備一個固定不變的開頭或結尾，而是會隨著應用程式持續運作，源源不絕地、以串流的形式持續產生。

**十二要素應用程式，完全不會去關心它自己所產生的輸出串流，最終要如何被路由（route）、或該被儲存到哪裡去。** 它**絕對不應該**主動嘗試去寫入、或管理任何形式的日誌檔案。取而代之的做法是，應用程式裡的每一個運作中的程式，都應該把自己的事件串流，以未經緩衝處理（unbuffered）的方式，直接寫到標準輸出（`stdout`）去。在開發階段，開發者可以直接在自己的終端機視窗裡，即時地觀察這一串輸出的內容，藉此了解應用程式當下的運作行為。

**而在 staging 環境或正式環境的部署裡，每一個程式各自所輸出的這些串流內容，則會被該應用程式所屬的執行環境（execution environment）給捕捉起來，統一彙整整理，再一併路由到一個（或多個）最終的目的地，以供查閱、或做長期的封存保留（archival）**——這整個彙整與路由的過程，並不是由應用程式自身來管理或監督的，而是完全交由這個應用程式所在的執行環境來負責。

坊間有許多開放原始碼的日誌路由工具，可以拿來完成這件事——例如 [Logplex](https://github.com/heroku/logplex)（隨附於 [Heroku](https://www.heroku.com/) 平台）、以及 [Fluentd](https://github.com/fluent/fluentd)（隨附於 [Google Cloud Platform](https://cloud.google.com/)）。

這樣經過彙整而成的事件串流，可以被匯出到一個檔案裡、也可以即時地被監看（watch）著、觀察最新產生的內容；此外，它也可以被送到一套日誌索引與分析系統裡，例如 [Splunk](http://www.splunk.com/)，或是一套像 [Hadoop/Hive](http://hive.apache.org/) 這類、適合用來做大規模資料分析的通用儲存系統。這些系統，帶來了非常強大的功能與威力，讓開發團隊能藉此觀察應用程式隨著時間推移、所呈現出來的行為趨勢，包括：

- 找出過去某個特定時間點，究竟發生了什麼重要的事件；
- 透過大規模的圖表視覺化呈現（例如每分鐘的請求數量隨時間變化），來觀察應用程式行為的長期趨勢；
- 根據事先設定好的一些門檻值（threshold）（例如：當每分鐘的錯誤發生數量，超過某個特定數值時），主動觸發警示（alerting）通知，讓相關人員能即時介入處理。

## XII. 管理型程式（Admin processes）

### 把管理／維運性質的任務，當作一次性（one-off）的程式來執行

**建置、發布、運行（build, release, run）**這三個階段，講述的是：程式碼是如何被轉換成一份份的（正式環境）部署的。**十二要素方法論**建議，任何跟這個應用程式相關的**一次性管理任務（one-off admin process）**，都應該要在一個**跟該應用程式一致（identical）**的環境裡來執行。

這類一次性管理任務，包含以下這兩大類：

- **一次性的腳本任務。** 舉例來說，一支用來執行資料庫遷移（migration）的腳本（例如 [Django](https://www.djangoproject.com/) 裡的 `manage.py migrate` 指令、Rails 裡的 `rake db:migrate` 指令）。這類任務，也包括那些偶爾才需要執行一次的維運腳本，用來完成一些特殊的維護性任務，例如：

  - 執行一支腳本，把資料庫裡幾筆錯誤資料，逐一修正過來；
  - 分析某段特定程式碼，過去一段時間以來，所產生的一些統計數據；

  這類一次性的腳本任務，理想的情況下，應該跟該應用程式的程式碼一起被保存起來（版本控制），置於同一份程式碼庫（repo）裡，以避免程式碼與腳本兩者之間，發生不同步（sync）的問題。

- **一次性的、互動式的 REPL shell。** 這指的是，開發者可以透過該程式語言直譯器（interpreter）所提供的一個「讀取—求值—輸出循環（read-eval-print loop, REPL）」，直接連進正式環境（或是其他任何一個部署環境）裡，執行一些互動式的操作與除錯——例如 Ruby 的 `rails console` 指令、Python 的 `python` 指令。

**這些一次性的管理任務，都應該跟該應用程式一般、長時間持續運作中的程式，在完全相同（identical）的環境裡執行。** 它們應該跟一般的程式一樣，套用同一份程式碼庫、同一份設定值，都是根據同一份發布版本（release）所建立而成的。管理型任務所使用的程式碼，也必須跟應用程式的一般程式一樣，遵循同樣的依賴隔離（dependency isolation）技術——舉例來說，如果一個 Ruby 網路程式使用 `bundle exec thin start` 這樣的指令來啟動的話，那麼一支用來執行資料庫遷移的一次性腳本，也理應要透過 `bundle exec rake db:migrate` 這樣的方式來執行，才符合十二要素方法論的原則。

在開發者本機端的部署環境裡，開發者可以直接透過終端機（shell），下達一次性的管理指令，來執行這類的管理任務。而在正式環境的部署裡，開發者則可以透過 SSH，或是該應用程式所屬執行環境（execution environment）所提供的其他等效指令執行機制，來下達並執行這類一次性的管理指令。

**十二要素方法論，也偏好那些原生內建有 REPL 直譯器（interpreter shell）的程式語言**，因為這樣的語言特性，讓開發者能更輕鬆地執行這類一次性的腳本任務。而在本機開發環境裡，開發者也應該要能透過像是 `bundle exec` 這類的指令，來取得一個具備一致性、能正確載入完整應用程式環境的 REPL shell。
